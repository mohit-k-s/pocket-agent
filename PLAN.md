# Devin-from-iPhone Plan (LAN-Only Web Pairing)

## 1) Goal
Build a mobile-friendly web app that lets an iPhone chat with a local Mac to run Devin tasks, with execution staying on the Mac (not Devin cloud VM).

## 2) Scope
- In scope:
  - Local Mac agent service
  - Mobile web/PWA chat UI
  - Pairing between phone and Mac
  - Streaming task output
  - LAN-only networking and security hardening
- Out of scope (v1):
  - Public internet access
  - Multi-user org features
  - App Store/native iOS app

## 3) High-Level Architecture
- **Mac Agent (local daemon):**
  - Runs on Mac
  - Manages Devin subprocess/task sessions
  - Exposes authenticated API for task creation + streaming
- **Web UI (PWA):**
  - Chat-style interface optimized for iPhone
  - Connects to Mac Agent over LAN
  - Supports text prompts + file/image uploads
- **Pairing + Auth:**
  - One-time pairing code generated by Mac Agent
  - Phone exchanges code for bearer token
  - Token stored in browser secure storage

## 4) Security Model (LAN-Only)
1. No router port forwarding / no UPnP exposure
2. Bind service to LAN interface only
3. Firewall allowlist (at least subnet, ideally paired device IP)
4. Pairing code with short TTL + one-time use
5. Bearer token required for all API calls
6. Strict CORS + Origin checks
7. Workspace path allowlist for task execution
8. No raw shell interpolation of user input
9. Audit logs for pair/task actions
10. Quick "unpair all devices" control

## 5) API Plan (v1)
- `POST /api/pair/start` → create short-lived pairing code
- `POST /api/pair/complete` → exchange code for token
- `GET /api/me` → verify auth + server status
- `POST /api/tasks` → create task from prompt
- `GET /api/tasks` → list tasks
- `GET /api/tasks/:id` → task status + metadata
- `GET /api/tasks/:id/stream` → SSE stream output
- `POST /api/tasks/:id/cancel` → stop task

## 6) Execution Plan for Devin
- v1: use `devin -p` for straightforward request/response tasks
- v1.1: add persistent sessions (tmux-backed) for follow-up chat continuity
- Keep default permission mode conservative unless explicitly changed

## 7) Implementation Milestones

### Milestone A — Project bootstrap
- Create repo structure:
  - `agent/` (Mac backend)
  - `web/` (PWA frontend)
  - `shared/` (types/contracts if needed)
- Add env/config templates
- Define workspace allowlist config

### Milestone B — Pairing + auth
- Implement pairing code generation/storage (TTL + one-time)
- Token issuance/verification
- Add token revocation endpoint and UI action

### Milestone C — Task lifecycle
- Implement task creation, status persistence, cancel flow
- Run Devin safely via subprocess APIs
- Capture stdout/stderr and normalize logs

### Milestone D — Streaming UX
- SSE endpoint for incremental output
- UI streaming chat bubbles + task states
- Reconnect handling on network drops

### Milestone E — Security hardening
- CORS/origin lock
- Rate limiting on auth/task endpoints
- Basic audit logging
- Firewall setup documentation/scripts (local)

### Milestone F — PWA polish
- Installable PWA metadata
- Mobile-first layout and keyboard-safe input
- Attachments (file/image upload)
- Basic settings screen (server URL, pair/unpair)

### Milestone G — Validation
- Test matrix:
  - iPhone Safari on same Wi‑Fi
  - task start/stream/cancel
  - unpair and token invalidation
  - restart agent and recover state
- Manual security checklist signoff

## 8) Suggested Tech Stack
- Backend: Node (Fastify/Express) **or** Python (FastAPI)
- Frontend: React + Vite (PWA plugin) **or** SvelteKit
- Streaming: SSE for simplicity in v1
- Storage: SQLite (tasks + pairing + tokens)

## 9) Open Decisions
1. Backend language preference (Node vs Python)
2. Data model depth for task history (ephemeral vs persisted)
3. Whether to include tmux persistent sessions in initial milestone
4. Preferred local TLS approach for LAN (self-signed vs local CA)

4. Implement single `devin -p` task path
5. Add streaming and mobile chat UI
